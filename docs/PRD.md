# 图片风格转换系统 - 产品需求文档 (PRD)

## 1. 文档信息
| 项目名称 | 图片风格转换系统（完整版） |
|---------|------------------------|
| 版本号   | v2.0                   |
| 创建日期 | 2026-06-16             |
| 状态     | 待开发                  |

## 2. 项目背景
当前系统已实现基于 Vue3 + Vite 的前端界面和 Python 后端核心图片风格转换功能，但缺少用户体系、数据存储和访问控制。本次迭代目标是将系统升级为多用户、可持久化存储的完整 Web 应用，支持用户注册/登录、图片上传转换记录、个人资源管理，并基于 JWT 进行接口权限校验。

## 3. 产品目标
- 建立完整的用户生命周期管理（注册、登录、信息修改、密码重置）
- 实现基于 JWT 的无状态身份认证及权限校验
- 用户能够将转换前后的图片持久化存储到云端或服务器，并可查看历史记录
- 提供个人文件管理能力（删除、下载、重复转换）
- 保证核心图片风格转换功能的流畅体验

## 4. 用户角色
| 角色     | 描述                         | 权限范围                                         |
|----------|------------------------------|--------------------------------------------------|
| 未登录用户 | 未注册或未登录的访客           | 仅可预览转换功能（体验模式，不保存记录）或禁止使用核心功能 |
| 普通用户   | 已注册并登录的个人用户         | 上传图片，执行风格转换，保存记录，管理自己的文件        |
| 管理员     | 拥有后台管理权限的用户（后续扩展） | 用户管理，系统监控，风格模型配置等（本次迭代可选）   |

## 5. 功能需求

### 5.1 用户模块
| 功能ID | 功能名称       | 描述                                                         | 优先级 |
|--------|----------------|--------------------------------------------------------------|--------|
| U-01   | 用户注册       | 提供用户名、邮箱、密码注册。密码需加密存储。                   | P0     |
| U-02   | 用户登录       | 用户名/邮箱 + 密码登录，后端返回 JWT Access Token 及 Refresh Token。 | P0     |
| U-03   | Token 续期     | 使用 Refresh Token 获取新的 Access Token。                   | P1     |
| U-04   | 用户登出       | 前端清除 Token，后端可选加入黑名单（简单实现可仅前端清除）。   | P1     |
| U-05   | 获取个人信息   | 查看个人资料（用户名、邮箱、注册时间、使用统计）。             | P1     |
| U-06   | 修改密码       | 验证旧密码后更新为新密码。                                   | P2     |
| U-07   | 重置密码（邮箱） | 通过注册邮箱发送重置链接（后续迭代）。                        | P3     |

### 5.2 图片风格转换模块（增强）
| 功能ID | 功能名称           | 描述                                                                 | 优先级 |
|--------|--------------------|----------------------------------------------------------------------|--------|

| S-02   | 选择风格模板       | 预设若干风格（如梵高、莫奈、动漫、素描等），支持扩展。                 | P0     |

| S-08   | 再次转换           | 基于历史记录的原始图片或结果图片，快速发起新的转换。                   | P2     |

### 5.3 存储与数据管理
| 功能ID | 功能名称           | 描述                                                                 | 优先级 |

## 6. 技术栈建议

| 层级     | 技术选型                                                                 |
|----------|--------------------------------------------------------------------------|
| 前端     | Vue3 + Vite + Pinia（状态管理） + Axios（HTTP） + Element Plus / Naive UI |
| 后端     | Python + FastAPI（或现有框架）                                           |
| 认证授权 | python-jose (JWT) + passlib (密码哈希)                                   |
| 任务队列 | Celery + Redis（异步处理风格转换耗时任务）                               |
| 数据库   | PostgreSQL 15+ 或 MySQL 8.0+                                            |
| 对象存储 | MinIO（自建）或 阿里云OSS / AWS S3                                       |
| 部署     | Docker Compose（前后端 + Redis + PostgreSQL + MinIO）                    |

## 8. API 设计概要（RESTful，JWT认证）

### 8.1 认证与用户
| 方法 | 端点                    | 说明                     | 认证 |
|------|------------------------|--------------------------|------|
| POST | `/api/auth/register`   | 用户注册                 | 否   |
| POST | `/api/auth/login`      | 登录，返回 token         | 否   |
| POST | `/api/auth/refresh`    | 刷新 token               | 是   |
| POST | `/api/auth/logout`     | 登出（可选黑名单）       | 是   |
| GET  | `/api/users/me`        | 获取当前用户信息         | 是   |
| PUT  | `/api/users/me/password` | 修改密码               | 是   |

### 8.2 图片转换
| 方法 | 端点                           | 说明                     | 认证 |
|------|--------------------------------|--------------------------|------|
| POST | `/api/convert/upload`          | 上传原始图片，返回 file_id | 是   |
| POST | `/api/convert/submit`          | 提交转换任务（传入 file_id, style） | 是 |
| GET  | `/api/convert/task/{task_id}`  | 查询任务状态及结果 URL    | 是   |
| GET  | `/api/convert/history`         | 获取用户历史记录（分页） | 是   |
| DELETE | `/api/convert/record/{id}`   | 删除某条记录及相关文件   | 是   |

> **说明**：实际开发中可采用异步方案，上传和提交合并为一个接口，后台返回 task_id，前端轮询或 WebSocket 获取结果。

## 9. 非功能需求

- **性能**：图片转换任务应控制在 30 秒内完成（视模型复杂度），并发任务数不超过 5 个/用户，使用队列削峰。
- **安全性**：
  - JWT Access Token 有效期 2 小时，Refresh Token 有效期 7 天。
  - 用户密码使用 bcrypt 或 argon2 加密。
  - 所有 API 需验证 Token（除注册/登录外）。
  - 上传文件需校验 MIME 类型，防止恶意文件。
  - 用户存储路径隔离：`/storage/user_{id}/...`，禁止路径遍历。
- **可用性**：系统应提供友好的错误提示（如存储不足、格式错误、任务超时）。
- **可扩展性**：风格模型应支持动态添加，通过数据库配置，无需重启后端服务。

## 10. 用户界面简要说明（与现有前端集成）

- **登录/注册页**：新增路由，表单验证。
- **个人中心**：展示用户信息、已用存储空间、历史记录列表（卡片或表格）。
- **转换页**：原有核心界面增强，增加“保存到我的作品”选项（默认开启）；未登录用户跳转登录或禁用转换按钮。
- **历史记录页**：展示所有转换记录，支持查看大图、下载、删除和“再次转换”。
- **顶部导航栏**：根据登录状态显示用户名/头像、登出按钮。

## 12. 风险与应对

| 风险                 | 应对措施                                         |
|----------------------|--------------------------------------------------|
| 图片转换耗时过长导致请求超时 | 采用异步任务队列 + 前端轮询进度，避免 HTTP 长连接 |
| 存储费用不可控         | 设置用户配额，定期清理未使用的临时文件，提供付费扩容选项（后期） |
| JWT 被盗用           | 使用短有效期 Access Token，配合 Refresh Token 以及 HTTPS 传输 |
| 并发转换导致服务器负载高 | 限制每个用户的并发任务数，使用 Redis 限流，动态扩展 worker |
